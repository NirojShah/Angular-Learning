# Input using Signals in Angular

# What is Input using Signals?

Angular introduced **Signal-based Inputs** to modernize component communication.

Before Signals:

Parent passed data using:

```ts
@Input()
```

Now Angular supports:

```ts
input()
```

This creates reactive input values automatically.

Old:

```text
Parent → Child (normal property)
```

New:

```text
Parent → Child (reactive signal)
```

Signal inputs automatically track updates.

---

# Why Signal Inputs?

Old `@Input()` works, but it has limitations:

* Not reactive by default
* Needs lifecycle hooks like `ngOnChanges`
* Harder to track changes
* More boilerplate

Signal Inputs solve:

* Reactive updates
* Cleaner syntax
* Better dependency tracking
* Works directly with computed/effect
* Better performance

---

# Old Way vs New Way

# Old Way (@Input)

Child:

```ts
import { Component, Input } from '@angular/core';

@Component({
  selector: 'app-child',
  template: `<p>{{ name }}</p>`
})
export class ChildComponent {
  @Input() name!: string;
}
```

Parent:

```html
<app-child [name]="parentName"></app-child>
```

Component:

```ts
parentName = "Niroj";
```

Flow:

```text
Parent → Child
```

Problem:

To detect changes:

```ts
ngOnChanges() {}
```

Needed.

---

# New Way (input())

Child:

```ts
import { Component, input } from '@angular/core';

@Component({
  selector: 'app-child',
  template: `<p>{{ name() }}</p>`
})
export class ChildComponent {
  name = input<string>();
}
```

Parent:

```html
<app-child [name]="parentName"></app-child>
```

Component:

```ts
parentName = "Niroj";
```

Flow:

```text
Parent → Signal Input → Child Template
```

Now `name` is reactive.

Read:

```ts
this.name()
```

No `ngOnChanges`.

---

# input() Signal Syntax

Basic:

```ts
property = input<Type>();
```

Example:

```ts
username = input<string>();
```

With default:

```ts
username = input("Guest");
```

Required:

```ts
username = input.required<string>();
```

Optional:

```ts
username = input<string>();
```

---

# Full Example: input() Signal

Parent Component:

```ts
import { Component, signal } from '@angular/core';

@Component({
  selector: 'app-root',
  templateUrl: './app.component.html'
})
export class AppComponent {
  userName = signal("Niroj");

  changeName() {
    this.userName.set("Angular");
  }
}
```

Parent Template:

```html
<app-child [name]="userName()"></app-child>

<button (click)="changeName()">Change</button>
```

---

Child Component:

```ts
import { Component, input } from '@angular/core';

@Component({
  selector: 'app-child',
  template: `<p>{{ name() }}</p>`
})
export class ChildComponent {
  name = input<string>();
}
```

Flow:

```text
Parent Signal → Child Signal Input → Child UI
```

Output:

```text
Niroj → Angular
```

Automatic update.

---

# Parent → Child using input() Signals

Example:

Parent:

```ts
title = signal("Angular Signals");
```

Template:

```html
<app-child [title]="title()"></app-child>
```

Child:

```ts
title = input<string>();
```

Template:

```html
<h1>{{ title() }}</h1>
```

Dependency:

```text
Parent title → Child title → Child Template
```

If parent updates:

```ts
title.set("Angular 21");
```

Child auto-updates.

No lifecycle hook.

---

# Using Computed with input()

Signal inputs work perfectly with computed.

Child:

```ts
name = input<string>();

upperName = computed(() => this.name().toUpperCase());
```

Template:

```html
<p>{{ upperName() }}</p>
```

Flow:

```text
Input Signal → Computed → Template
```

Very powerful.

---

# Using Effect with input()

Child:

```ts
name = input<string>();

constructor() {
  effect(() => {
    console.log(this.name());
  });
}
```

Whenever parent changes:

```text
Effect runs automatically
```

No `ngOnChanges`.

---

# Child → Parent (Old Way: EventEmitter)

Traditional child-to-parent communication.

Child:

```ts
import { Output, EventEmitter } from '@angular/core';

@Output() save = new EventEmitter<string>();

sendData() {
  this.save.emit("Hello Parent");
}
```

Template:

```html
<button (click)="sendData()">Send</button>
```

Parent:

```html
<app-child (save)="receiveData($event)"></app-child>
```

Parent Component:

```ts
receiveData(data: string) {
  console.log(data);
}
```

Flow:

```text
Child → EventEmitter → Parent
```

---

# Child → Parent (New Way: Signals)

Modern way using model().

Child:

```ts
import { model } from '@angular/core';

name = model('');
```

Template:

```html
<input [(ngModel)]="name">
```

Parent:

```html
<app-child [(name)]="parentName"></app-child>
```

Parent Component:

```ts
parentName = "Niroj";
```

Flow:

```text
Parent ↔ Child (shared reactive signal)
```

No EventEmitter needed.

This creates two-way signal binding.

---

# EventEmitter vs Signal Output

| EventEmitter           | Signal Output     |
| ---------------------- | ----------------- |
| Manual emit            | Auto reactive     |
| More boilerplate       | Cleaner syntax    |
| One-way child → parent | Two-way possible  |
| Uses events            | Uses shared state |
| Older pattern          | Modern pattern    |

---

# When to use what?

Use `@Input()`:

* Older projects
* Simple non-reactive inputs

Use `input()`:

* Modern Angular
* Signal-based architecture
* Reactive component communication

Use `EventEmitter`:

* Trigger events

Use `model()`:

* Shared state / two-way communication

---

# input.required()

Used when parent must pass value.

Example:

```ts
name = input.required<string>();
```

If parent does not pass:

Error occurs.

Useful for mandatory inputs.

---

# input() with Alias

Old:

```ts
@Input('user-name') name!: string;
```

New:

```ts
name = input('', { alias: 'user-name' });
```

Parent:

```html
<app-child [user-name]="username"></app-child>
```

---

# input() vs @Input()

| @Input()          | input()            |
| ----------------- | ------------------ |
| Decorator         | Function           |
| Normal property   | Signal             |
| Needs ngOnChanges | Auto reactive      |
| Less optimized    | Fine-grained       |
| Not tracked       | Dependency tracked |

---

# Common Mistakes

# Forgetting ()

Wrong:

```ts
this.name
```

Correct:

```ts
this.name()
```

Signal input is a signal.

---

# Using ngOnChanges unnecessarily

Wrong:

```ts
ngOnChanges() {}
```

Usually unnecessary with signal input.

Use:

```ts
effect()
```

---

# Mutating input directly

Wrong:

```ts
this.name.set("Hello");
```

Input signals are readonly.

Parent owns the value.

---

# Best Practices

* Use `input()` for modern components
* Use `computed()` for derived values
* Use `effect()` for side effects
* Use `input.required()` for mandatory data
* Use `model()` for two-way communication

# Interview Questions

## What is input() in Angular?

Signal-based input API for parent-to-child communication.

---

## Difference between @Input and input()?

@Input:
Normal property.

input():
Reactive signal.

---

## Can input() be used with computed?

Yes.

Very common.

---

## Can input() be updated inside child?

No.

It is readonly.

---

## What replaces ngOnChanges in signal inputs?

```ts
effect()
```

---

## What is model()?

Signal-based two-way binding between parent and child.

---

## Difference between EventEmitter and model()?

EventEmitter:
Event-based.

model():
Shared reactive state.

---

## When should we use input.required()?

When parent must pass data.
