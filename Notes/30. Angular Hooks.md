# Angular Lifecycle Hooks - Complete Documentation

# What are Lifecycle Hooks?

Lifecycle Hooks are **special methods provided by Angular** that allow you to execute code at different stages of a component's lifecycle.

A component goes through different phases:

* Creation
* Initialization
* Change Detection
* Rendering
* Destruction

Angular calls these hooks automatically at the appropriate time.

Think of them as **events in a component's life**.

---

# Why are Lifecycle Hooks Needed?

Without Lifecycle Hooks:

* No way to know when a component is created.
* Difficult to fetch data.
* Hard to initialize resources.
* Memory leaks may occur.
* Cleanup becomes difficult.

With Lifecycle Hooks:

* Initialize data.
* Fetch APIs.
* Access DOM.
* Detect input changes.
* Clean subscriptions.
* Prevent memory leaks.

---

# Angular Component Lifecycle

```text
Application Starts
        │
        ▼
Constructor
        │
        ▼
ngOnChanges
        │
        ▼
ngOnInit
        │
        ▼
ngDoCheck
        │
        ▼
ngAfterContentInit
        │
        ▼
ngAfterContentChecked
        │
        ▼
ngAfterViewInit
        │
        ▼
ngAfterViewChecked
        │
        ▼
Component Running
        │
        ▼
ngOnDestroy
```

> **Note:** `ngOnChanges()` runs before `ngOnInit()` **only if the component has one or more `@Input()`/`input()` values**.

---

# Types of Lifecycle Hooks

Angular provides the following lifecycle hooks:

| Hook                  | Purpose                              |
| --------------------- | ------------------------------------ |
| constructor           | Creates the component instance       |
| ngOnChanges           | Detects input property changes       |
| ngOnInit              | Initializes the component            |
| ngDoCheck             | Custom change detection              |
| ngAfterContentInit    | Content projection initialized       |
| ngAfterContentChecked | Projected content checked            |
| ngAfterViewInit       | Component view initialized           |
| ngAfterViewChecked    | Component view checked               |
| ngOnDestroy           | Cleanup before component destruction |

---

# 1. Constructor

The constructor is a TypeScript feature, **not an Angular lifecycle hook**.

Purpose:

* Create the component instance.
* Dependency Injection.
* Initialize simple values.

Example:

```ts
constructor(private http: HttpClient) {
  console.log('Constructor Called');
}
```

Do **not** make API calls inside the constructor.

---

# Constructor vs ngOnInit

| Constructor                   | ngOnInit                                     |
| ----------------------------- | -------------------------------------------- |
| TypeScript feature            | Angular lifecycle hook                       |
| Runs first                    | Runs after Angular initializes the component |
| Used for Dependency Injection | Used for initialization logic                |
| Avoid API calls               | Best place for API calls                     |

---

# 2. ngOnChanges

Triggered whenever an `@Input()` or `input()` property changes.

Implements:

```ts
import { OnChanges } from '@angular/core';

export class ChildComponent implements OnChanges {

  ngOnChanges() {
    console.log('Input Changed');
  }

}
```

Using `SimpleChanges`:

```ts
import {
  OnChanges,
  SimpleChanges
} from '@angular/core';

ngOnChanges(changes: SimpleChanges) {

  console.log(changes);

}
```

Example:

Parent changes:

```ts
userName = 'John';
```

↓

```ts
userName = 'Alice';
```

↓

```text
ngOnChanges()
```

Called automatically.

---

# 3. ngOnInit

Most commonly used lifecycle hook.

Runs **once** after Angular initializes the component.

Implements:

```ts
import { OnInit } from '@angular/core';

export class HomeComponent implements OnInit {

  ngOnInit() {
    console.log('Component Initialized');
  }

}
```

Common Uses:

* API calls
* Initialize forms
* Read route parameters
* Load user data
* Initialize signals

Example:

```ts
ngOnInit() {

  this.loadUsers();

}
```

---

# 4. ngDoCheck

Runs during every Angular change detection cycle.

```ts
import { DoCheck } from '@angular/core';

export class AppComponent implements DoCheck {

  ngDoCheck() {

    console.log('Checking Changes');

  }

}
```

Use Cases:

* Custom change detection
* Detect object mutations
* Advanced performance tuning

Use carefully because it runs frequently.

---

# 5. ngAfterContentInit

Runs once after Angular projects external content using `<ng-content>`.

Example:

Child:

```html
<div>

  <ng-content></ng-content>

</div>
```

Hook:

```ts
import { AfterContentInit } from '@angular/core';

export class CardComponent
implements AfterContentInit {

  ngAfterContentInit() {

    console.log('Content Projected');

  }

}
```

Runs once.

---

# 6. ngAfterContentChecked

Runs after Angular checks projected content.

```ts
ngAfterContentChecked() {

  console.log('Content Checked');

}
```

Runs many times during change detection.

---

# 7. ngAfterViewInit

Runs once after Angular initializes the component's view and child views.

Most commonly used with:

* `@ViewChild`
* `@ViewChildren`

Example:

```ts
import {
  ViewChild,
  ElementRef,
  AfterViewInit
} from '@angular/core';

export class HomeComponent
implements AfterViewInit {

  @ViewChild('inputBox')
  input!: ElementRef;

  ngAfterViewInit() {

    this.input.nativeElement.focus();

  }

}
```

HTML:

```html
<input #inputBox>
```

---

# 8. ngAfterViewChecked

Runs after Angular checks the component's view.

```ts
ngAfterViewChecked() {

  console.log('View Checked');

}
```

Runs frequently.

Avoid heavy logic.

---

# 9. ngOnDestroy

Runs before Angular destroys the component.

Used for cleanup.

```ts
import {
  OnDestroy
} from '@angular/core';

export class HomeComponent
implements OnDestroy {

  ngOnDestroy() {

    console.log('Destroyed');

  }

}
```

Common Uses:

* Unsubscribe Observables
* Clear timers
* Remove event listeners
* Disconnect WebSocket
* Stop intervals

Example:

```ts
subscription!: Subscription;

ngOnDestroy() {

  this.subscription.unsubscribe();

}
```

# Lifecycle Hooks and Signals

Signals work seamlessly with lifecycle hooks.

Example:

```ts
count = signal(0);

ngOnInit() {

  this.count.set(10);

}
```

Using `effect()`:

```ts
constructor() {

  effect(() => {

    console.log(this.count());

  });

}
```

In many cases, Signals reduce the need for hooks like `ngDoCheck` because Angular automatically tracks signal dependencies.

---

# Best Practices

* Use the constructor only for Dependency Injection.
* Use `ngOnInit()` for initialization.
* Use `ngOnDestroy()` for cleanup.
* Avoid expensive operations in `ngDoCheck()`.
* Avoid changing component state inside `ngAfterViewChecked()`.
* Prefer Signals for reactive state instead of manual change detection where possible.

---

# Common Mistakes

## API Call in Constructor

Wrong:

```ts
constructor() {

  this.http.get(...);

}
```

Correct:

```ts
ngOnInit() {

  this.http.get(...);

}
```

---

## Forgetting Cleanup

Wrong:

```ts
this.subscription =
this.service.getData().subscribe();
```

Correct:

```ts
ngOnDestroy() {

  this.subscription.unsubscribe();

}
```

---

## Heavy Logic in ngDoCheck()

Wrong:

```ts
ngDoCheck() {

  // Complex calculations

}
```

Since `ngDoCheck()` runs often, heavy logic can hurt performance.

# Lifecycle Hooks Summary

| Hook                  | Runs                   | Common Use                    |
| --------------------- | ---------------------- | ----------------------------- |
| constructor           | Component creation     | Dependency Injection          |
| ngOnChanges           | Input changes          | React to input updates        |
| ngOnInit              | Once                   | API calls, initialization     |
| ngDoCheck             | Every change detection | Custom change detection       |
| ngAfterContentInit    | Once                   | `<ng-content>` initialization |
| ngAfterContentChecked | Every content check    | Monitor projected content     |
| ngAfterViewInit       | Once                   | `@ViewChild`, DOM access      |
| ngAfterViewChecked    | Every view check       | View monitoring               |
| ngOnDestroy           | Once before destroy    | Cleanup resources             |

# Interview Questions

## What are Lifecycle Hooks?

Lifecycle Hooks are Angular methods that allow developers to execute code at specific stages of a component's lifecycle.

---

## Which hook is used for API calls?

`ngOnInit()`.

---

## Difference between Constructor and ngOnInit?

The constructor is a TypeScript feature used for Dependency Injection, while `ngOnInit()` is an Angular lifecycle hook used for initialization.

---

## Which hook runs first?

The `constructor` runs first.

If the component has inputs:

```text
Constructor

↓

ngOnChanges

↓

ngOnInit
```

Otherwise:

```text
Constructor

↓

ngOnInit
```

---

## Which hook is called when an @Input() changes?

`ngOnChanges()`.

---

## Which hook is used with @ViewChild?

`ngAfterViewInit()`.

---

## Which hook is used with ng-content?

`ngAfterContentInit()`.

---

## Which hook is used for cleanup?

`ngOnDestroy()`.

---

## Why should ngDoCheck() be used carefully?

Because it runs during every Angular change detection cycle and can affect performance if expensive operations are performed.

---

## Can Signals replace Lifecycle Hooks?

Not completely.

Signals reduce the need for manual change detection and some reactive logic, but hooks like `ngOnInit()` and `ngOnDestroy()` are still important for initialization and cleanup.
