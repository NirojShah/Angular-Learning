# Route Guards - CanDeactivate in Angular

# What is CanDeactivate?

`CanDeactivate` is a Route Guard that decides whether a user is allowed to leave the current route.

Unlike `CanActivate`, which runs **before entering** a route, `CanDeactivate` runs **before leaving** the current route.

It is mainly used to prevent users from accidentally losing unsaved changes.

Flow:

```text
User clicks another route
        │
        ▼
Angular Router
        │
        ▼
CanDeactivate Guard Executes
        │
   ┌────┴────┐
   │         │
 true      false
   │         │
   ▼         ▼
Leave     Stay on
Page      Current Page
```

---

# Why do we need CanDeactivate?

Imagine a user is filling out a registration form.

```text
Name: Niroj

Email: niroj@email.com

Password: ********
```

The user accidentally clicks another page before saving.

Without `CanDeactivate`:

```text
User Leaves Page

↓

All entered data is lost.
```

With `CanDeactivate`:

```text
User Tries to Leave

↓

Warning Appears

↓

"Do you really want to leave?"

↓

User decides whether to stay or leave.
```

This greatly improves the user experience.

# Real-World Use Cases

Use `CanDeactivate` for:

* Registration forms
* Profile editing
* Payment forms
* Blog editors
* Product creation
* Survey forms
* Any page containing unsaved changes

---

# CanActivate vs CanDeactivate

| CanActivate                         | CanDeactivate                      |
| ----------------------------------- | ---------------------------------- |
| Before entering a route             | Before leaving a route             |
| Protects secured pages              | Prevents accidental navigation     |
| Checks authentication/authorization | Checks unsaved changes             |
| Runs before component loads         | Runs before component is destroyed |

# Creating a CanDeactivate Guard

Generate the guard using Angular CLI.

```bash
ng generate guard guards/can-deactivate
```

Short command:

```bash
ng g guard guards/can-deactivate
```

Generated file:

```text
guards/
   can-deactivate.guard.ts
```

---

# Step 1 - Create an Interface

It is a good practice to define an interface that components can implement.

```ts
export interface CanComponentDeactivate {
  canDeactivate: () => boolean;
}
```

This ensures every protected component provides a `canDeactivate()` method.

---

# Step 2 - Create the Guard

```ts
import { CanDeactivateFn } from '@angular/router';
import { CanComponentDeactivate } from './can-component-deactivate';

export const canDeactivateGuard:
CanDeactivateFn<CanComponentDeactivate> =

(component) => {

  return component.canDeactivate();

};
```

Explanation:

The guard simply asks the current component:

```text
Can I leave this page?
```

The component decides.

---

# Step 3 - Component Example

```ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-profile',
  templateUrl: './profile.html'
})
export class ProfileComponent {

  hasUnsavedChanges = true;

  canDeactivate(): boolean {

    if (!this.hasUnsavedChanges) {
      return true;
    }

    return confirm(
      'You have unsaved changes. Leave this page?'
    );

  }

}
```

If the user confirms:

```text
true
```

Navigation continues.

Otherwise:

```text
false
```

Navigation is cancelled.

---

# Step 4 - Use Guard in Routes

```ts
import { Routes } from '@angular/router';
import { canDeactivateGuard } from './guards/can-deactivate.guard';

export const routes: Routes = [

  {
    path: 'profile',
    component: ProfileComponent,
    canDeactivate: [
      canDeactivateGuard
    ]
  }

];
```

Now Angular checks the guard before leaving the Profile page.

# Example with Reactive Forms

```ts
form = new FormGroup({
  username: new FormControl('')
});

canDeactivate() {

  if (this.form.dirty) {

    return confirm(
      'Unsaved changes detected. Leave?'
    );

  }

  return true;

}
```

Angular checks whether the form is dirty.

If yes:

Ask for confirmation.

---

# Example using Signals

```ts
import { signal } from '@angular/core';

hasUnsavedChanges = signal(false);

updateData() {
  this.hasUnsavedChanges.set(true);
}

save() {
  this.hasUnsavedChanges.set(false);
}

canDeactivate() {

  if (this.hasUnsavedChanges()) {

    return confirm(
      'Leave without saving?'
    );

  }

  return true;

}
```

Signals work perfectly with `CanDeactivate`.

# Returning Different Types

A `CanDeactivate` guard can return:

```ts
true
```

Allow navigation.

```ts
false
```

Cancel navigation.

```ts
Promise<boolean>
```

Useful for asynchronous checks.

```ts
Observable<boolean>
```

Useful for API-based confirmation.

Example:

```ts
return dialog.afterClosed();
```

---

# Multiple CanDeactivate Guards

```ts
{
  path: 'profile',
  component: ProfileComponent,
  canDeactivate: [
    guardOne,
    guardTwo
  ]
}
```

Angular executes guards in order.

If any guard returns `false`, navigation stops.

---

# Best Practices

* Use `CanDeactivate` only when necessary.
* Keep confirmation messages simple.
* Use Reactive Forms' `dirty` property when possible.
* Keep business logic outside the guard.
* Let the component decide whether it has unsaved changes.
* Use Signals or Reactive Forms to track changes.

---

# Common Mistakes

## Always Returning true

Wrong:

```ts
canDeactivate() {
  return true;
}
```

This completely defeats the purpose of the guard.

---

## Putting Form Logic Inside the Guard

Wrong:

```ts
// Read form controls
// Validate form
// Check dirty state
```

The component already knows its state.

The guard should simply ask the component.

---

## Forgetting to Reset Unsaved State

Wrong:

```ts
save() {
  // Save Data
}
```

Correct:

```ts
save() {

  // Save Data

  this.hasUnsavedChanges = false;

}
```

Or with Signals:

```ts
this.hasUnsavedChanges.set(false);
```

# Interview Questions

## What is CanDeactivate?

A Route Guard that decides whether the current route can be exited before Angular navigates away.

---

## Why do we use CanDeactivate?

To prevent users from accidentally losing unsaved changes.

---

## Difference between CanActivate and CanDeactivate?

| CanActivate     | CanDeactivate      |
| --------------- | ------------------ |
| Before entering | Before leaving     |
| Authentication  | Unsaved changes    |
| Protects routes | Protects user data |

---

## When should we use CanDeactivate?

Whenever users can edit or create data.

Examples:

* Forms
* Profile editing
* Payment
* Survey
* Blog editor

---

## Can CanDeactivate return asynchronous values?

Yes.

It supports:

* `Promise<boolean>`
* `Observable<boolean>`

---

## Can Reactive Forms be used with CanDeactivate?

Yes.

The `dirty` property is commonly used.

---

## Can Signals be used with CanDeactivate?

Yes.

Signals can easily track whether there are unsaved changes.

---

## Who decides whether navigation should continue?

The component.

The guard simply asks the component by calling its `canDeactivate()` method.

---

## Can multiple CanDeactivate guards be used?

Yes.

Angular executes them sequentially.
